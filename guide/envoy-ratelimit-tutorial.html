Architecture

Every request flows through Envoy before it reaches your app. Envoy calls the rate limit service via gRPC on each request; the service checks Redis counters and returns ALLOW or OVER_LIMIT.

import jakarta.servlet.http.HttpServletRequest;
import lombok.extern.slf4j.Slf4j;

    }

Configure rate limiting rules

Create the directory structure below. The rate limit service scans for YAML files inside the config/ subdirectory at startup and loads every file it finds. You can split rules across multiple files — all are merged into one ruleset.

ratelimit/
config/
limits.yaml  ← create this

The complete file first, then a field-by-field breakdown:

YAML ratelimit/config/limits.yaml 
domain: my_domain
descriptors:

  - key: remote_address
    rate_limit:
      unit: MINUTE
      requests_per_unit: 10

  - key: path
    value: /hello-world
    rate_limit:
      unit: MINUTE
      requests_per_unit: 5

① domain

domain is a namespace string that groups all the rules in this file. The rate limit service uses it as the first part of every Redis counter key, e.g. my_domain_remote_address_1.2.3.4. Its only job is to prevent collisions — if two services share the same Redis instance, giving them different domains keeps their counters separate.

⚠
This value must be copied exactly into envoy.yaml under the rate limit filter's domain field. Case-sensitive. A mismatch means Envoy sends descriptors to a domain with no matching rules — requests will pass through unchecked.

② descriptors

A descriptor is a rule. Each one says: "when Envoy sends me a key (optionally with a value), apply this limit." The rate limit service receives the descriptors Envoy assembled from the request (see Step 5 for how Envoy builds them) and looks for the best match here.

There are two types of match:

  • key-only match — matches any descriptor with that key, regardless of value. Used for broad rules like "any IP".
  • key + value match — matches only when both key and value match exactly. Used for targeted rules like "the /hello-world path specifically".

Both descriptor types in the config are independent. They are not an if/else — both are evaluated for every request that carries a matching descriptor. If the request hits multiple rules, the most restrictive one wins (i.e. the one that is closest to its limit).

③ Rule 1 — per-IP limit

YAML Rule 1 breakdown
- key: remote_address
  # "key" is the descriptor key that Envoy must send for this
  # rule to fire. This matches the remote_address action
  # configured in envoy.yaml — Envoy sends:
  # { key: "remote_address", value: "1.2.3.4" }
  # No "value" field here means this rule matches ANY IP address.
  # Each unique IP gets its own counter in Redis.

  rate_limit:
    unit: MINUTE
    # The time window for the counter. Options:
    # SECOND — resets every second (aggressive, low-overhead)
    # MINUTE — resets every 60 seconds (most common)
    # HOUR   — resets every 3600 seconds
    # DAY    — resets every 86400 seconds
    # The window is rolling from the first request, not
    # aligned to the wall clock (not "top of the minute").

    requests_per_unit: 10
    # Maximum requests allowed within one "unit" window.
    # Once this counter reaches 10 for a given IP, all
    # further requests from that IP return 429 until the
    # window resets. The counter is stored as a single
    # Redis key with a TTL equal to the window duration.

④ Rule 2 — per-path limit

YAML Rule 2 breakdown
- key: path
  # Must match the descriptor_key defined in the
  # request_headers action in envoy.yaml. Envoy sends:
  # { key: "path", value: "/hello" }

  value: /hello-world
  # Because "value" is specified, this rule ONLY fires when
  # Envoy sends exactly { key: "path", value: "/hello" }.
  # A request to /actuator/health sends { key: "path",
  # value: "/actuator/health" } — no match, no limit applied.
  # The match is exact string equality, not a prefix or regex.

  rate_limit:
    unit: MINUTE
    requests_per_unit: 5
    # Tighter limit than the IP rule above: only 5/min
    # to /hello across ALL callers combined (not per-IP).
    # This is a global counter for the path, not scoped
    # to individual users. If you want per-user-per-path
    # limits, nest descriptors (see the extensions section).

⑤ How Redis keys are formed

Each matching descriptor becomes a Redis key in the format domain_key_value_unit. Understanding this helps when you inspect Redis directly to debug unexpected behaviour.

Descriptor receivedRedis keyTTL
remote_address = 127.0.0.1my_domain_remote_address_127.0.0.160s
path = /hello-worldmy_domain_path_/hello-world60s

You can inspect these live while the stack is running:

BASH Inspect Redis counters 
docker compose exec redis redis-cli keys "*my_domain*"
docker compose exec redis redis-cli get "my_domain_path_/hello-world"

⑥ Nested descriptors (advanced)

You can nest descriptors to create compound rules — for example, limit per-user-per-path rather than just per-path globally:

YAML Nested descriptor example
descriptors:
  - key: user_id
    # outer key — matches on user_id from a request header
    descriptors:
      - key: path
        value: /hello-world
        rate_limit:
          unit: MINUTE
          requests_per_unit: 20
    # This rule only fires when BOTH conditions are met:
    # the descriptor contains a user_id key AND a
    # path = /hello-world key. Each user_id gets its own
    # 20/min counter for /hello.

① static_resources

static_resources is the top-level key that holds everything Envoy knows at startup — its listeners (what to accept) and its clusters (where to forward). The alternative is dynamic_resources, which lets a control plane push config changes at runtime (used in service meshes like Istio). For a local setup like this, static is simpler and sufficient.

② listener

A listener is a port that Envoy opens and waits for connections on. Think of it as the front door. You can define multiple listeners (e.g. one for HTTP, one for HTTPS) but here one is enough.

YAML listener socket
address: 0.0.0.0      # bind to all network interfaces inside the container
port_value: 8090      # the port Docker exposes to the host

0.0.0.0 means "accept connections on any network interface" — necessary inside Docker so the container can receive traffic from the host. If you used 127.0.0.1 instead, only processes inside the same container could reach it.

③ filter_chains and the HttpConnectionManager

Every TCP connection that lands on the listener is processed by a filter chain — an ordered list of network-level filters. For HTTP traffic there is always exactly one filter here: the HttpConnectionManager (HCM). The HCM is Envoy's HTTP engine. It parses HTTP/1.1 and HTTP/2 frames and hands the request to its own inner pipeline of HTTP filters.

YAML HCM config fields
stat_prefix: ingress_http
# A label prepended to every metric this HCM emits.
# e.g. http.ingress_http.downstream_rq_total
# Visible at localhost:9901/stats — change it to something
# meaningful if you run multiple listeners.

use_remote_address: true
# Tells Envoy to trust the real client IP from the TCP
# connection rather than the X-Forwarded-For header that
# arrived with the request. Without this, the remote_address
# descriptor in rate_limits would always see Envoy's own IP
# instead of the actual caller's IP, making IP-based limiting
# useless.

④ route_config — virtual hosts and routes

The route_config block is Envoy's routing table. It maps incoming requests to upstream clusters.

YAML route_config breakdown
virtual_hosts:
  - name: local_service
    domains: ["*"]
    # domains controls which Host header values this virtual host
    # matches. "*" means any host — fine for a single-service
    # setup. In production you'd set this to your domain name
    # e.g. ["api.example.com"] to route different domains to
    # different clusters from the same listener.

    routes:
      - match:
          prefix: "/"
          # prefix: "/" matches every request path.
          # You can be more specific: use "path: /api/v1/hello-world"
          # for exact match, or "safe_regex" for a regex.
          # Routes are evaluated top-to-bottom; first match wins.
        route:
          cluster: spring_boot_service
          # The cluster name must match a key under the
          # top-level "clusters" list exactly.
          rate_limits:
            - actions:
                - remote_address: {}
            - actions:
                - request_headers:
                    header_name: ":path"
                    descriptor_key: path
          # CRITICAL: rate_limits must be nested inside route: (the
          # RouteAction), not at the route entry level or on the
          # virtual_host. Placing it anywhere else silently skips
          # the rate limit check — no error, no 429, just nothing.

⑤ rate_limits actions — what Envoy sends to the rate limit service

This is the bridge between Envoy and the rate limit service. For every request, Envoy executes each action block and assembles a list of descriptors — key-value pairs — which it sends to the rate limit service via gRPC. The rate limit service looks up each descriptor against limits.yaml and returns ALLOW or OVER_LIMIT for each one.

YAML rate_limits actions explained
rate_limits:
  - actions:
      - remote_address: {}
    # Action type "remote_address" extracts the caller's IP
    # (populated because use_remote_address: true is set above)
    # and sends the descriptor: { key: "remote_address", value: "1.2.3.4" }
    # The rate limit service matches this against the
    # "key: remote_address" rule in limits.yaml.

  - actions:
      - request_headers:
          header_name: ":path"
          descriptor_key: path
    # Action type "request_headers" reads the value of the
    # named header and uses descriptor_key as the key name.
    # ":path" is an HTTP/2 pseudo-header that holds the URL
    # path — Envoy populates it for HTTP/1.1 too.
    # For a request to /hello this sends:
    # { key: "path", value: "/hello" }
    # The rate limit service matches this against the
    # "key: path, value: /hello" rule in limits.yaml.
    #
    # If the path doesn't match any descriptor in limits.yaml
    # (e.g. /actuator/health) the rate limit check passes
    # through with no limit applied.
ℹ
Each actions entry is evaluated independently. For a single request to /hello-world from IP 1.2.3.4, Envoy makes two descriptor checks: one for the IP rule (10/min) and one for the path rule (5/min). The request is blocked if either limit is exceeded — whichever is hit first.

⑥ http_filters — the rate limit filter and the router

HTTP filters run in order on every request inside the HCM. There are always at least two: the rate limit filter (your custom logic) and the router (always last — it forwards the request to the cluster).

YAML ratelimit filter fields
domain: my_domain
# REQUIRED. Must exactly match the domain in limits.yaml.
# Namespaces all Redis counters — keeps multiple apps using
# the same Redis instance from colliding.

request_type: external
# Which requests to rate limit.
# "external" = real client traffic only (x-envoy-internal not set).
# "internal" = Envoy-to-Envoy calls only.
# "both"     = all requests. Defaults to "both" if omitted.

failure_mode_deny: false
# What to do if the rate limit service is unreachable.
# false = fail open  — requests pass through (safe for dev).
# true  = fail closed — requests get 500 (safer for production).

rate_limited_as_resource_exhausted: true
# Affects gRPC callers only (HTTP callers always get 429).
# true  = return gRPC status RESOURCE_EXHAUSTED when rate limited.
# false = return gRPC status UNAVAILABLE.
# Set to true — RESOURCE_EXHAUSTED is the correct gRPC semantic.

enable_x_ratelimit_headers: DRAFT_VERSION_03
# Adds X-RateLimit-Limit, X-RateLimit-Remaining, and
# X-RateLimit-Reset headers to every response so clients
# can see their quota. DRAFT_VERSION_03 is the current standard.
# Omit or set to OFF to disable these headers.

rate_limit_service:
  grpc_service:
    envoy_grpc:
      cluster_name: rate_limit_service
  # REQUIRED. Points to the cluster for the rate limit gRPC service.
  # envoy_grpc uses Envoy's built-in gRPC client — simpler than
  # google_grpc which requires the C-core gRPC library.

  transport_api_version: V3
  # Must be V3. V2 is deprecated.

⑦ clusters — upstreams

A cluster is a named set of endpoints Envoy can forward requests to. Think of each cluster as a backend service. Envoy load-balances across endpoints within a cluster. Here there are two: your Spring Boot app and the rate limit gRPC service.

YAML spring_boot_service cluster
name: spring_boot_service
# Arbitrary name. Must match what is referenced in route.cluster above.

connect_timeout: 1s
# How long Envoy waits when opening a new TCP connection to
# this upstream. If the app is slow to start (e.g. on first
# request) you may need to raise this.

type: STRICT_DNS
# How Envoy resolves the upstream address.
# STRICT_DNS: resolves the hostname on every DNS TTL refresh
#             and uses all returned A records as endpoints.
#             Correct for Docker Compose where service names
#             are DNS-resolvable within the shared network.
# STATIC:     you supply a hardcoded IP — not useful here.
# LOGICAL_DNS: like STRICT_DNS but only uses one IP at a time.
# EDS:        dynamic endpoint discovery from a control plane.

address: spring_boot
# The Docker Compose service name. Docker's internal DNS
# resolves "spring_boot" to the container's IP automatically
# because both services are on the same "rl_net" network.

port_value: 8080
# The port your Spring Boot app listens on inside the container
# (set in application.yml). This is the container-internal
# port, NOT the host-mapped port.
YAML rate_limit_service cluster
name: rate_limit_service
# Must match cluster_name in the ratelimit filter above.

connect_timeout: 0.25s
# Much shorter than the app cluster because this is an
# internal sidecar service. If it doesn't respond in 250ms
# something is seriously wrong.

http2_protocol_options: {}
# THIS IS MANDATORY for gRPC. gRPC is built on HTTP/2.
# Without this Envoy defaults to HTTP/1.1 and every gRPC
# call silently fails. The empty braces {} just enable the
# option with defaults — no further configuration needed.
# This is the single most common source of confusion when
# setting up Envoy + rate limiting for the first time.

address: ratelimit
# The Docker Compose service name for the rate limit container.

port_value: 8081
# The gRPC port the rate limit service listens on internally.
# This is its container-internal port, not the host-mapped
# 8082 port in docker-compose.yml.

⑧ admin

The admin block enables Envoy's built-in management HTTP server on port 9901. It is entirely separate from the listener and serves no user traffic — only diagnostic endpoints. You can reach it at http://localhost:9901 in your browser.

YAML admin
address: 0.0.0.0
# Bind to all interfaces so it's reachable from the host
# via the port mapping in docker-compose.yml.

port_value: 9901
# Standard Envoy admin port. Useful endpoints:
#   /stats         — all metrics as plain text
#   /clusters      — health of each upstream cluster
#   /config_dump   — the full resolved running config as JSON
#   /listeners     — active listeners
# Never expose this port publicly in production.
⚠
The domain value in envoy.yaml and limits.yaml must match exactly — including case. A mismatch means Envoy sends descriptors to a domain with no matching rules.

  # ── Your Spring Boot app ──────────────────────────────
  spring_boot:
    build: .
    ports:
      - "8080:8080"   # optional: direct access bypassing Envoy
    networks:
      - rl_net

  # ── Redis (counter storage) ───────────────────────────
  redis:
    image: redis:7-alpine
    networks:
      - rl_net

  # ── Lyft rate limit service ───────────────────────────
  # This is the open-source rate limit service originally
  # written by Lyft, now maintained under the Envoy Proxy
  # organisation. It exposes a gRPC endpoint that Envoy
  # calls on every request (configured in envoy.yaml).
  # It reads your limits.yaml rules, checks the counters
  # in Redis, and returns ALLOW or OVER_LIMIT to Envoy.
  ratelimit:
    image: envoyproxy/ratelimit:master
        # The official image from the envoyproxy org on Docker Hub.
    # "master" always pulls the latest build from the main
    # branch. Pin to a digest (e.g. @sha256:abc123) in
    # production so builds are reproducible.
    command: /bin/ratelimit
        # The entrypoint binary inside the image. This starts
    # the gRPC server and loads the config from the volume.
    environment:
      LOG_LEVEL: debug
            # Verbosity of logs. "debug" prints every descriptor
      # check and its result — useful while developing.
      # Change to "warn" or "error" in production to reduce noise.
      REDIS_SOCKET_TYPE: tcp
            # How to connect to Redis. "tcp" means a standard
      # TCP socket over the network. The alternative is
      # "unix" for a Unix domain socket, only useful if
      # Redis is co-located on the same host filesystem.
      REDIS_URL: redis:6379
            # Host:port of the Redis server. "redis" resolves to
      # the Redis container via Docker's internal DNS because
      # both services share the rl_net network. 6379 is
      # Redis's default port.
      USE_STATSD: "false"
            # Whether to emit metrics to a StatsD agent.
      # Disabled here — no StatsD container in this stack.
      # Set to "true" and add STATSD_HOST/STATSD_PORT if
      # you want to ship metrics to Datadog, Grafana, etc.
      RUNTIME_ROOT: /data
            # The base directory the service looks in for config.
      # Combined with RUNTIME_SUBDIRECTORY, it constructs
      # the full path: /data/ratelimit/config/limits.yaml
      RUNTIME_SUBDIRECTORY: ratelimit
            # Appended to RUNTIME_ROOT. The service then looks for
      # YAML files inside a "config/" folder within this
      # subdirectory. The "config/" level is mandatory —
      # files placed directly in /data/ratelimit/ are ignored.
      RUNTIME_WATCH_ROOT: "false"
            # Whether to watch the config directory for file changes
      # and hot-reload rules without restarting. "false" here
      # because the volume is static. Set to "true" if you
      # want to edit limits.yaml and have changes apply live.
      RUNTIME_IGNOREDOTFILES: "true"
            # Skip any files starting with "." in the config
      # directory (e.g. .gitkeep, editor swap files).
      # Prevents the service from trying to parse non-YAML
      # files and crashing on startup.
      GRPC_MAX_CONCURRENT_STREAMS: "1000000"
            # Maximum number of in-flight gRPC streams the server
      # will accept simultaneously. One stream per request
      # means this is effectively your max in-flight checks.
      # The default is much lower — raising it prevents
      # backpressure under high traffic.
    volumes:
      - ./ratelimit:/data/ratelimit
      # Mounts your local ratelimit/ directory (containing
      # config/limits.yaml) into the container at /data/ratelimit.
      # This is how the service reads your rules without
      # needing to bake them into a custom Docker image.
      # Changes to limits.yaml require a container restart
      # unless RUNTIME_WATCH_ROOT is set to "true".
    depends_on:
      - redis
      # Ensures Redis starts before the rate limit service.
      # Note: depends_on only waits for the container to
      # start, not for Redis to be ready to accept connections.
      # The rate limit service retries on startup so this
      # is fine in practice.
    networks:
      - rl_net

  # ── Envoy proxy (public entry point) ─────────────────
  # Envoy is the only container that receives traffic from
  # the outside world (your browser, Postman, curl).
  # It intercepts every request, calls the rate limit service
  # over gRPC to check the counter, then either forwards the
  # request to Spring Boot or returns 429 — all before your
  # application code runs. Spring Boot never sees blocked
  # requests and has no rate-limiting logic of its own.
  envoy:
    image: envoyproxy/envoy:v1.29-latest
        # Pinned to v1.29. Envoy follows a rapid release cycle
    # (~3 releases/year). The tag format is vMAJOR.MINOR-latest.
    # Avoid "latest" — it can silently break your config when
    # a new major version ships with breaking API changes.
    volumes:
      - ./envoy/envoy.yaml:/etc/envoy/envoy.yaml
      # Mounts your local envoy.yaml into the container at the
      # path Envoy reads by default. Like the rate limit config,
      # this avoids building a custom image just to embed config.
      # A restart is required to pick up changes to this file.
    ports:
      - "8090:8090"
            # Maps host port 8090 → container port 8090 (the
      # listener defined in envoy.yaml). This is the only
      # port you should send Postman/curl traffic to.
      # Format is "HOST:CONTAINER".
      - "9901:9901"
            # Envoy's built-in admin HTTP server. Exposes diagnostic
      # endpoints: /stats, /clusters, /config_dump, /listeners.
      # Open http://localhost:9901 in a browser to inspect
      # Envoy's live state. Never expose this publicly.
    command: envoy -c /etc/envoy/envoy.yaml --log-level info
        # -c points Envoy at your config file.
    # --log-level controls verbosity: trace, debug, info, warn,
    # error, critical. "info" logs each request and its outcome.
    # Use "debug" to see the rate limit gRPC calls in detail.
    depends_on:
      - spring_boot
      - ratelimit
      # Envoy starts after both upstreams are up. If Spring Boot
      # or the rate limit service isn't ready yet, Envoy will
      # retry the connection — it does not crash on startup if
      # an upstream is temporarily unreachable.
    networks:
      - rl_net

  rl_net:

Test with Postman

All requests should go through port 8090 (Envoy), not 8080 directly.

Basic request

  1. Create a new request
    Method: GET   URL: http://localhost:8090/api/v1/hello-world
  2. Send it
    You should get a 200 OK with {"message":"Hello!","client_ip":"..."}
  3. Check the response headers
    You should see x-ratelimit-limit, x-ratelimit-remaining, and x-ratelimit-reset — Envoy adds these automatically because enable_x_ratelimit_headers: DRAFT_VERSION_03 is set.

Understanding the response headers

Here's what a real response looks like after the first request:

BASH Example response headers
x-ratelimit-limit: 5, 10;w=60, 5;w=60
x-ratelimit-remaining: 4
x-ratelimit-reset: 13
x-envoy-upstream-service-time: 4
server: envoy
HeaderExample valueWhat it means
x-ratelimit-limit 5, 10;w=60, 5;w=60 Two rules matched — the IP rule (10/min) and the path rule (5/min). The leading 5 is the effective limit: the most restrictive rule wins. w=60 means a 60-second rolling window.
x-ratelimit-remaining 4 Requests left in the current window before hitting the limit. Starts at limit−1 after the first request and counts down to 0.
x-ratelimit-reset 13 Seconds until the counter resets. This is a rolling window from the first request — not a fixed clock boundary like "top of the minute".
x-envoy-upstream-service-time 4 Milliseconds your Spring Boot app took to respond. Useful for spotting latency from the app vs. Envoy overhead.

Trigger the rate limit

Use Postman's Runner to send the request repeatedly and watch for a 429:

  1. Open Collection Runner
    Click the ▶ icon next to your collection, or go to Runner in the sidebar
  2. Set iterations to 10
    Set delay to 0ms so requests fire as fast as possible
  3. Run
    After the 5th request to /api/v1/hello-world you should see 429 Too Many Requests. The x-ratelimit-remaining header will show 0 on the last allowed request.
ℹ
After a 429 the counter doesn't reset immediately — x-ratelimit-reset tells you exactly how many seconds to wait. The IP rule (10/min) and path rule (5/min) have independent counters in Redis, so the path counter may reset before the IP counter.

Common extensions

Per-user rate limiting

Key on an authenticated user ID header instead of (or in addition to) IP address:

YAML envoy.yaml — add to rate_limits.actions 
- actions:
    - request_headers:
        header_name: "x-user-id"
        descriptor_key: user_id
YAML limits.yaml — add a user_id descriptor 
- key: user_id
  rate_limit:
    unit: HOUR
    requests_per_unit: 1000

Shadow mode (dry run)

Observe rate limit decisions without actually blocking traffic — useful for testing limits before enforcing them:

YAML envoy.yaml — add to the ratelimit filter config 
shadow_mode: true   # counters increment, but no 429s

Expose limit headers to the Spring Boot app

If you want your app to read the rate limit headers (e.g. to add them to an API response body), inject them in IntelliJ as a filter:

JAVA RateLimitFilter.java 
@Component
public class RateLimitFilter implements Filter {

    @Override
    public void doFilter(ServletRequest req, ServletResponse res,
                         FilterChain chain) throws IOException, ServletException {
        HttpServletRequest  http = (HttpServletRequest)  req;
        HttpServletResponse out  = (HttpServletResponse) res;

        String remaining = http.getHeader("x-ratelimit-remaining");
        if (remaining != null) {
            out.setHeader("X-App-RateLimit-Remaining", remaining);
        }

        chain.doFilter(req, res);
    }
}
⚠

Troubleshooting

Rate limit headers missing — no x-ratelimit-* in response
The most common cause is rate_limits being placed at the wrong level in envoy.yaml. It must be nested inside route: (the RouteAction), not at the route entry level or on the virtual host. If it's at the wrong level, Envoy silently ignores it — no error, no 429, just no headers. Also check that upstream_rq_total on the rate_limit_service cluster is incrementing via curl -s http://localhost:9901/stats | grep rate_limit_service. If it's zero, the filter isn't firing at all.
Envoy exits with "no such field: rate_limits" on the route
rate_limits is a field on the RouteAction proto, not on the Route proto. It must be indented under route:, not at the same level as match: and route:.
rate_limit_service upstream_rq_total stays at 0
Envoy isn't calling the rate limit service at all. Check: (1) rate_limits is correctly nested inside route:, (2) request_type is set to both — using external can cause requests from localhost/Docker to be treated as internal and skipped, (3) the @type on the rate limit filter is envoy.extensions.filters.http.ratelimit.v3.RateLimit — not the HCM type.
Envoy exits with "no such field: stage"
stage is a valid field in the current proto but was removed then restored across patch versions. If you see this error, simply remove stage: 0 — it defaults to 0 anyway and is not needed for a single-filter setup.
Envoy exits with "no such field: failure_mode_deny"
failure_mode_deny is a valid top-level field per the official proto. If you see this error, check your indentation — it must be at the same level as domain and rate_limit_service, not nested inside rate_limit_service.
Envoy exits with "no such field: rate_limited_resource_exhausted"
The field name is rate_limited_as_resource_exhausted — the word as is required. Check your envoy.yaml for the typo.
Envoy exits with "no such field: domain" (or rate_limit_service)
Both are top-level fields on the RateLimit filter. If Envoy rejects them, the @type URL is likely wrong or misindented. It must be exactly type.googleapis.com/envoy.extensions.filters.http.ratelimit.v3.RateLimit, and all fields must be indented under typed_config, not under name.
All requests return 429 immediately
The domain value in envoy.yaml and limits.yaml must match exactly — including case. A mismatch means Envoy sends descriptors to a domain with no matching rules, so requests pass through unchecked. Check both files and restart with docker compose restart envoy ratelimit.
Rate limit service doesn't load config
The volume mount in docker-compose.yml maps ./ratelimit to /data/ratelimit. The service then looks for files at RUNTIME_ROOT/RUNTIME_SUBDIRECTORY/config/*.yaml — which is /data/ratelimit/config/limits.yaml. The config/ subdirectory is required. Run docker compose exec ratelimit ls /data/ratelimit/config/ to verify.
gRPC connection errors in Envoy logs
The http2_protocol_options: {} field on the rate_limit_service cluster is mandatory — gRPC uses HTTP/2 and Envoy defaults to HTTP/1.1. Without it every gRPC call fails. Re-check your envoy.yaml and restart Envoy.
Spring Boot container exits immediately
The jar must exist before Docker builds the image. Run ./mvnw package -DskipTests in IntelliJ terminal first, then docker compose up --build. The Dockerfile COPY instruction will fail if the jar is missing.
Limits reset faster or slower than expected
The unit in limits.yaml defines a rolling window from the first request, not a wall-clock boundary. MINUTE means 60 seconds from the first request in the window, not "resets at the top of the minute". This is expected behavior.
Postman shows no x-ratelimit-* headers
Make sure you are sending to port 8090 (Envoy) not 8080 (Spring Boot directly). Envoy adds those headers; the app knows nothing about them. Also confirm the rate limit filter is active with curl -s http://localhost:9901/config_dump | grep ratelimit.
